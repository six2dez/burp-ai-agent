<objective>
REL-03: Route retryable HTTP failures (429 / 5xx) through `CircuitBreaker.recordFailure` in all four HTTP backends so the breaker actually sees upstream overload and can fail-fast. Today each backend calls `recordFailure()` only inside its connection-error `catch`, never on the `!resp.isSuccessful` HTTP-status branch â€” so a server returning a stream of 429/503 never trips the breaker. This is the exact gap that produced Phase 14's deferred WR-05 (Anthropic, and the OpenAiCompatible analog, both skipped the recording).

Fix it ONCE: add a shared `isRetryableHttpStatus` / `recordHttpFailureIfRetryable` helper to `HttpBackendSupport` and call it identically from all four backends' failure branches. NVIDIA + Perplexity delegate to OpenAiCompatible (covered automatically); BurpAi uses native Burp AI with no HTTP (correctly excluded).

Purpose: Prevent unbounded retry/load amplification against a struggling upstream (DoS-amplification resilience control), and eliminate the four-near-identical-copies drift that caused WR-05.
Output: Extended `HttpBackendSupport.kt`, one added line in each of 4 backends, and `HttpBackendCircuitFailureTest.kt` proving the breaker opens behaviorally after repeated 429/5xx.
</objective>
