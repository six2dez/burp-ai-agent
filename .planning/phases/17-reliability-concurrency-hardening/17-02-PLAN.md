<objective>
REL-01: Make `ChatPanel`'s four session maps (`sessionPanels`, `sessionStates`, `sessionsById`, `sessionDrafts`) safely EDT-confined. Today the backend `onComplete` callback (`ChatPanel.kt:570`) runs on the backend connection's executor thread — NOT the EDT — and from there `maybeExecuteToolCall` (`:2038`) reads `sessionPanels[sessionId]`/`sessionsById[sessionId]` (`:2049–2050`) and calls `panel.addMessage(...)` (`:2074`, `:2096`), concurrently with EDT mutations from `createSession`/`deleteSession`/`restore`. That is a real data race on plain `linkedMapOf`s.

Per the locked CONTEXT decision: confine to the EDT (assert + `invokeLater`), do NOT convert to thread-safe collections (which would not fix the off-EDT `panel.addMessage` Swing call and would hide read-modify-write races). Preserve insertion order (the maps are `linkedMapOf` for ordering). Create a zero-dependency local `@GuardedBy` annotation (JCIP/jsr305 are deliberately off-classpath) and add `-ea` to the test task so the EDT assertion actually fires in CI.

Purpose: Eliminate the concrete UI data race that can corrupt session state or throw ConcurrentModificationException during multi-step tool chains.
Output: `util/GuardedBy.kt` (new), EDT-confinement edits in `ChatPanel.kt`, `jvmArgs("-ea")` in `build.gradle.kts`, and a `ChatPanelConcurrencyTest` that is the real SC1 gate.
</objective>
