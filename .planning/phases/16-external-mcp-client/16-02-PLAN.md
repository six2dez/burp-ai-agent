<objective>
Define the `ExternalMcpServerConfig` data model and `ExternalMcpTransport` enum; extend `McpSettings` with the `externalMcpServers` list; bump `AgentSettings` to schema v5 with per-field encrypted token load/save; enable the Wave 0 migration test stubs.

Purpose: Establishes the persistence layer so Plan 03 (client manager) can read configs with plaintext `bearerToken` values — the encryption boundary lives entirely here. Per-field SecretCipher encryption mirrors exactly how `ollamaApiKey`, `anthropicApiKey`, and all other API-key fields are handled (encrypt-on-save, decrypt-on-load, `ENC1:` prefix at rest). No new crypto code.

Crypto contract (D-SC4 / BLOCKER 2 fix): The in-memory `ExternalMcpServerConfig.bearerToken` is ALWAYS plaintext. Encryption happens only inside `saveExternalMcpServers()` before writing to prefs; decryption happens only inside `loadExternalMcpServers()` after reading from prefs. Callers of `loadExternalMcpServers()` — including Plan 03's ExternalMcpClientManager — receive the plaintext token directly and MUST NOT call `cipher.decrypt()` on it again.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Burp Preferences → AgentSettings | Serialized external server list (with per-field encrypted token) crosses from JVM preferences storage into application memory; token is decrypted per-field on load |
| AgentSettings → SecretCipher | Per-field token encryption: `cipher.encrypt(config.bearerToken, KEY_EXT_MCP_SERVERS)` on save; `cipher.decrypt(stored.bearerToken, KEY_EXT_MCP_SERVERS)` on load; plaintext only exists in memory transiently during these operations |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
