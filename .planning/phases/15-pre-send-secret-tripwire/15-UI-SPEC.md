# Phase 15 ‚Äî UI Design Contract: Pre-Send Secret Tripwire

> Visual and interaction contract for Phase 15 (Pre-Send Secret Tripwire). Generated by
> gsd-ui-researcher. Pending verification by gsd-ui-checker.
>
> **This is a minimal delta on the Phase 13 contract.** It does NOT re-specify the design system,
> spacing, typography, or the existing `ContextPreviewDialog` survived-secret banner ‚Äî all of that is
> inherited verbatim from `13-UI-SPEC.md` and remains authoritative. Phase 15 changes exactly two
> things in ONE existing component:
>
> 1. The Phase 13 `SubtleNotice` survived-secret banner **escalates from WARN to RISK** when a
>    pre-send tripwire match is present in the post-redaction payload (the highlight, SC5).
> 2. The existing `Send` action in `ContextPreviewDialog.confirm()` becomes a **warn-with-confirmation
>    gate** ("Send anyway / Cancel") ‚Äî and only that ‚Äî when a tripwire match is present (SC2 / SC5).
>
> Everything else (tokens, builders, light/dark rules, the un-migrated-dialog flag FLAG-13-02) is
> unchanged from Phase 13. No new design tokens. No new component. No new modal.

**Out of UI scope (documented, NOT designed here):**

- **Non-interactive paths** ‚Äî `scanner/PassiveAiScanner.kt` and `mcp/McpToolContext.redactIfNeeded()`
  on a tripwire hit **audit-log and proceed**; they show **no dialog and no UI**. SC4's "fires on all
  three paths" means *detect + audit-log* on all three; the **confirmation gate is ChatPanel-only**
  (via this dialog). These two paths have zero UI surface ‚Äî they are not part of this contract.
- **Detection logic** ‚Äî the tripwire detector (`SecretShapes` known-shape matches + Shannon-entropy
  heuristic) is an AWT-free backend object in/near the `redact` package. Its name, location, entropy
  threshold, and min token length are backend/planner concerns (CONTEXT.md ¬ß Claude's Discretion).
  This contract only consumes its boolean result: *is a tripwire match present in this payload?*
- **Audit logging** ‚Äî the "send anyway" allowlist event (session ID + truncated entropy score, never
  the raw value) is written via the existing `AuditLogger.logEvent(...)`. That is a backend behaviour
  (SC3), not a UI element. It produces **no on-screen output in this dialog**.

## Delta 1 ‚Äî Banner Escalation (WARN ‚Üí RISK) (SC5: highlight)

The Phase 13 banner reused `SubtleNotice.Level.WARN` (amber) for *survived-shape* advisories. Phase 13
explicitly **reserved `Level.RISK` (red) for "the Phase 15 tripwire's warn-with-confirmation"**
(13-UI-SPEC ¬ß Color, FLAG-13-03). Phase 15 cashes in that reservation.

### Three-state banner (supersedes Phase 13's two-state banner in this dialog)

| Banner state | Condition (on the post-redaction payload) | `SubtleNotice.Level` | Resulting palette (existing tokens) |
|--------------|-------------------------------------------|----------------------|-------------------------------------|
| **Clean** | No survived shape, no tripwire match | hidden (`hideNotice()`) | ‚Äî |
| **Advisory** (Phase 13 behaviour) | A known secret shape survived, but it is **not** a tripwire-gating match | `Level.WARN` (amber) | `subtleWarning` bg + `accentWarn` strip |
| **Tripwire** (NEW, SC5) | A tripwire match is present (known shape escalated to gating, or a high-entropy token) | `Level.RISK` (red) | `subtleDanger` bg + `accentDanger` strip |

**Rationale for RISK on the tripwire:** the tripwire signal is strictly higher-severity than the
advisory ‚Äî it *gates the Send* (Delta 2). Red (`RISK`) is the correct severity to mark a signal that
adds a confirmation step, and Phase 13 pre-allocated it for exactly this. `RISK` is an already-active
level (used by `SettingsPanel` privacy-OFF and `ChatPanel`), so this introduces no new visual.

> **Implementation note (not a new token):** `Level.RISK` maps via `UiTheme.Colors.accentDanger ‚Üí
> statusCrashed` and `subtleDanger`. No `Color()` literal is added; the banner inherits whatever
> `SubtleNotice` RISK already renders, exactly as Phase 13 inherited WARN (parallel to FLAG-13-03).

> **Simplification allowed (FLAG-15-01):** If the planner/executor finds the Advisory-vs-Tripwire
> distinction does not arise in practice (e.g. every gating match is also a surviving shape, so the
> banner is effectively just "clean ‚Üí RISK when gated"), collapsing to a two-state
> clean/RISK banner is acceptable ‚Äî the load-bearing requirements are (a) a *visible* highlight when a
> tripwire match is present (SC5) and (b) it reads as higher severity than a plain advisory. The
> three-state table is the preferred, most-informative rendering; the two-state collapse is a
> permitted equivalent, not a deviation.

### Banner copy (tripwire / RISK state)

Sentence case; names shape **category** only (never the raw matched value ‚Äî it is a suspected secret;
echoing it would re-expose it in the UI, same discipline as Phase 13). English-only (AGENTS.md).

| Variant | Copy |
|---------|------|
| Tripwire match ‚Äî high-entropy token (no named shape) | "A high-entropy value that may be a secret survived redaction. Review before sending." |
| Tripwire match ‚Äî named shape(s) | "A value matching a known secret shape ({shapes}) survived redaction. Review before sending." |
| Tripwire match ‚Äî mixed / multiple | "{N} values that may be secrets survived redaction ({shapes}). Review before sending." |

`{shapes}` = comma-joined human-readable category names from `SecretShapes` (e.g. "AWS access key",
"OpenAI key", "JWT", "high-entropy hex key"). `{N}` = count. The Phase 13 advisory copy ("‚Ä¶Review
before sending.") is retained for the WARN/Advisory state unchanged.

## Delta 2 ‚Äî The Confirmation Gate (SC1 / SC2: warn-with-confirmation, never hard-stop)

`ContextPreviewDialog.confirm()` today returns `Boolean` from a single
`JOptionPane.showOptionDialog(... options = ["Send", "Cancel"], default = "Cancel")`; `Send` ‚Üí `true`.

Phase 15 makes **the existing Send decision a warn-with-confirmation gate, and ONLY when a tripwire
match is present**. No new modal is introduced ‚Äî the gate reuses the dialog's own confirmation step.

### Gate behaviour

| Tripwire match present? | Dialog behaviour | Action labels | Return contract |
|-------------------------|------------------|---------------|-----------------|
| **No** | Unchanged from today | `Send` / `Cancel` | `Send` ‚Üí `true`; `Cancel` ‚Üí `false` |
| **Yes** | Same preview + RISK banner (Delta 1); the affirmative action carries the warn prompt and is relabeled to make "proceed past a secret" explicit | **`Send anyway`** / `Cancel` | `Send anyway` ‚Üí `true` (= the allowlist action, audit-logged in the backend); `Cancel` ‚Üí `false` |

**Two acceptable renderings of the warn prompt (executor's discretion ‚Äî both satisfy SC1/SC2):**

1. **Relabel-in-place (preferred, smallest delta):** When a tripwire match is present, the existing
   `showOptionDialog` affirmative option becomes `"Send anyway"`, the default selected option stays
   `Cancel`, and the warn sentence is shown in the RISK banner (Delta 1) ‚Äî the banner *is* the prompt.
   The dialog message already reads as a review-before-send; the red banner + "Send anyway" label make
   the warn-with-confirmation explicit without a second pop-up.
2. **Secondary confirm pop-up:** Keep the first dialog's `Send` as-is; on `Send` *with a tripwire
   match present*, show one follow-up `JOptionPane.showConfirmDialog` carrying the warn copy
   (below) with `Send anyway` / `Cancel`. Only `Send anyway` returns `true`.

Either way: **the affirmative is never the default focus**, and **Cancel is always available** ‚Äî there
is no hard-stop and no silent block (CONTEXT.md: "warn-with-confirmation, never a hard-stop"; SC2: a
legitimate base64 fuzz payload must be dismissible to proceed).

### Gate copy (the warn prompt ‚Äî NEW)

Seeded from CONTEXT.md SC1 ("This payload appears to contain a high-entropy value ‚Äî send anyway?").
Sentence case; problem-then-choice; English-only.

| Element | Copy |
|---------|------|
| Gate prompt (high-entropy / generic) | "This payload appears to contain a high-entropy value that may be a secret. Send anyway?" |
| Gate prompt (named shape) | "This payload appears to contain a value matching a known secret shape ({shapes}). Send anyway?" |
| Affirmative action (NEW label) | "Send anyway" |
| Negative action | "Cancel" (unchanged ‚Äî existing option) |

If rendering style 1 (relabel-in-place) is used and the prompt sentence already lives in the RISK
banner, the gate prompt above MAY be omitted from a second dialog to avoid duplicate copy ‚Äî the banner
sentence + "Send anyway" label together constitute the warn-with-confirmation.

### What stays the same (do NOT change)

- The dialog title "Review context before sending to AI", the privacy-mode line, the prompt preview,
  and the post-redaction context body ‚Äî all unchanged.
- The **clean-path** experience: when no tripwire match is present the dialog is byte-for-byte today's
  behaviour (`Send` / `Cancel`, `Send` ‚Üí `true`). The gate is additive and conditional.
- The single call site `ChatPanel.startSessionFromContext()` (`ui/ChatPanel.kt:299`) keeps consuming a
  `Boolean`; `false` still routes to the existing "cancelled by user" path. **No call-site signature
  break is required by this contract** ‚Äî if the detector result must reach the dialog, prefer the
  existing in-dialog self-scan precedent (the dialog already calls `SecretShapes.findSurviving(...)`
  on `contextJson` itself) so `confirm()`'s public shape can stay backward-compatible. Whether the
  dialog calls the tripwire detector directly or receives a precomputed result is a planner/executor
  decision, not a design decision.

## States

| State | Where | Presentation |
|-------|-------|--------------|
| Clean (no shape, no tripwire) | ContextPreviewDialog | Banner hidden; `Send` / `Cancel`; unchanged from today |
| Advisory (survived shape, not gating) | ContextPreviewDialog | `SubtleNotice` **WARN** (amber); `Send` / `Cancel` (Phase 13 behaviour) |
| Tripwire match present | ContextPreviewDialog | `SubtleNotice` **RISK** (red) + gate: `Send anyway` / `Cancel`, default `Cancel` |
| Tripwire on non-interactive path | PassiveAiScanner / McpToolContext | **No UI** ‚Äî audit-log + proceed (out of UI scope) |

No empty/loading state ‚Äî this is an inline change to a dialog the user already sees only at send time.

Inherited Phase 13 strings (banner advisory copy, "Send", "Cancel", privacy-mode hints) are unchanged.
NEW strings introduced by Phase 15:

| Element | Copy | Source |
|---------|------|--------|
| Primary CTA (this phase) | **"Send anyway"** ‚Äî the affirmative action when a tripwire match is present (= the audit-logged allowlist action) | This spec, seeded by CONTEXT.md SC1 |
| Gate prompt (generic) | "This payload appears to contain a high-entropy value that may be a secret. Send anyway?" | CONTEXT.md SC1 |

**Destructive action in this phase:** "Send anyway" is the one warn-gated action (sending a payload
that may contain a live secret). Confirmation approach = the warn-with-confirmation gate above, with

proceed is audit-logged by the backend (SC3), but no destructive-red "type to confirm" pattern is
used ‚Äî a one-click `Send anyway` against a non-default-focused button is the agreed friction level.

## Flags (Non-Blocking)

- **FLAG-15-01:** Banner may collapse from the three-state (clean / WARN advisory / RISK tripwire)
  table to a two-state (clean / RISK) banner if the Advisory case never materially differs from the
  Tripwire case in practice. The load-bearing requirements are a *visible* highlight on a tripwire
  match (SC5) reading as higher severity than a plain advisory. Three-state is preferred; two-state is
  a permitted equivalent.

- **FLAG-15-02:** The warn prompt may be rendered either by relabeling the existing
  `showOptionDialog` affirmative to "Send anyway" with the sentence carried in the RISK banner
  (preferred, smallest delta), or via a secondary `showConfirmDialog`. Both satisfy SC1/SC2 provided
  Cancel stays the default focus and no hard-block is introduced. Avoid showing the same warn sentence
  twice (banner + second dialog) ‚Äî pick one home for it.

- **FLAG-15-03:** Whether the tripwire detector result reaches the dialog by the dialog calling the
  detector directly (mirroring the existing `SecretShapes.findSurviving` self-scan at line 60) or via
  a precomputed parameter is left to the planner/executor. The contract only requires that the gate +
  RISK banner fire iff a tripwire match is present in the post-redaction payload, and that the public
  `confirm()` boolean contract its single caller relies on is not gratuitously broken.

- **FLAG-15-04 (carried from FLAG-13-02):** `ContextPreviewDialog` remains un-migrated to
  `DesignTokens` (raw `BorderLayout(8,8)`, `Dimension(780,560)`, `Box.createVerticalStrut`). Phase 15
  does NOT migrate it; only the conditional banner level + action relabel are added. Full migration
  stays a future UI-debt candidate.
