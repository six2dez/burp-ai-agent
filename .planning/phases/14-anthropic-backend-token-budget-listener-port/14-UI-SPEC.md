# Phase 14 — UI Design Contract: Anthropic Backend + Token Budget + Listener Port

> Visual and interaction contract for Phase 14 (Anthropic Backend + Token Budget + Listener Port).
> Generated by gsd-ui-researcher. Pending verification by gsd-ui-checker.
>
> This contract IS the authoritative design source for the small, additive UI touch points in this
> phase. All styling MUST come from the Phase 9 design system (`ui/design/DesignTokens.kt` +
> `ui/design/Components.kt`) — no inline `Color()` / `Font()` / spacing literals in panel code.
>
> **Phase 14 is mostly non-UI backend work** (the Anthropic Messages API backend, SSE streaming,
> `MontoyaHttpTransport` routing, `CircuitBreaker`, token accounting on `TokenTracker`, and the MCP
> `proxy_http_history` listener-port filter). The UI surface is intentionally small: **two new form
> regions in the existing backend/settings panels + one reused advisory banner in the chat panel.**
> CAP-03 (listener-port filter) is a pure MCP-tool parameter with **no UI** and is noted only for
> completeness.

## Context

**Tech stack:** Kotlin + Swing (JVM 21). No web framework. No shadcn / Tailwind / CSS.
**Platform constraint (ADR-2, locked):** Plain Swing + `javax.swing.UIManager` only. No JavaFX,
no FlatLaf, no third-party L&F. The design system is a Kotlin module, not a CSS file.
**Design system source:** Phase 9 — `DesignTokens.kt` + `Components.kt` in `ui/design/`.
**Light/dark:** Burp switches theme via `UIManager.setLookAndFeel()`. All tokens resolve dynamically
from UIManager keys at call time (computed `get` properties) — never cached.

**Scope of this contract — three touch points (all additive):**

1. **Anthropic backend card** in `ui/panels/BackendConfigPanel.kt` — a new `buildAnthropicPanel()`
   card added to the existing `CardLayout` (`cards.add(buildAnthropicPanel(), "anthropic")`),
   structurally **identical** to `buildOpenAiCompatPanel()` / `buildNvidiaNimPanel()` /
   `buildPerplexityPanel()`. Rows: editable **Model** (`JTextField`, defaults to `claude-sonnet-4-6`)
   and **API key** (`JPasswordField`, masked, encrypted on save via Phase 12 `SecretCipher`). No Base
   URL / Extra headers / Timeout rows are required by SC1–SC3 (the Anthropic endpoint is fixed); see
   Touch Point 1 for the exact row set. (CAP-01)

2. **Token-budget threshold inputs** in `ui/SettingsPanel.kt` — two integer fields (warn threshold,
   hard cap) in a new **"Token budget"** `sectionPanel` placed inside the existing Passive AI Scanner
   settings region (the budget governs the passive scanner's pause behavior, so it belongs next to
   the scanner controls — see Touch Point 2 for the placement decision). Both default to `0`
   (= unlimited / off). (CAP-04)

3. **Non-blocking token-budget banner** in `ui/ChatPanel.kt` — a reused `SubtleNotice` added at
   `BorderLayout.NORTH` of the existing `chatContainer`. Shown `Level.WARN` (amber) when the warn
   threshold is crossed, escalated to `Level.RISK` (red) when the hard cap fires and the passive
   scanner is paused. Hidden when no threshold is set or neither threshold is crossed. Mirrors the
   Phase 13 survived-secret `SubtleNotice` reuse pattern. (CAP-04)

**Everything else in Phase 14 is non-UI:** `backends/anthropic/AnthropicBackend.kt` +
`AnthropicBackendFactory.kt`, `BackendRegistry` registration, SSE parsing, `MontoyaHttpTransport`
routing, `CircuitBreaker`, `AgentSettings.anthropicApiKey` + `SecretCipher` wiring, `TokenTracker`
budget accounting, `PassiveAiScanner` pause-at-cap logic, and the MCP `proxy_http_history`
`listener_port` parameter (CAP-03, no UI).

### 60/30/10 Split (unchanged)

- **60% dominant:** `Colors.surface` — backend card, Token-budget section, and chat container backgrounds.
- **30% secondary:** `Colors.inputBackground` + `Colors.border` — the four new input fields (Model,
  API key, warn threshold, hard cap).
- **10% accent (`Colors.primary`):** focus rings + the existing "Test connection" button only.

### Chat token-budget banner color (via `SubtleNotice.Level`)

The banner reuses `SubtleNotice` and picks one of its three existing levels by **threshold state**:

| Banner state | `SubtleNotice.Level` | Resulting accent (existing tokens) | Rationale |
|--------------|----------------------|------------------------------------|-----------|
| Warn threshold crossed, hard cap not yet reached (or cap = 0/off) | `Level.WARN` | `UiTheme.Colors.subtleWarning` bg + `UiTheme.Colors.accentWarn` accent strip | Advisory — interactive chat fully usable; nothing has been paused. Amber. |
| Hard cap reached → passive scanner paused | `Level.RISK` | `UiTheme.Colors.subtleDanger` bg + `UiTheme.Colors.accentDanger` accent strip | A guardrail has actively fired and a subsystem (the passive scanner) is now paused. This is the strongest state the phase produces. Red. |
| No threshold set, or neither threshold crossed | banner hidden (`hideNotice()`) | — | Off-by-default; the banner is silent until a user-configured threshold is crossed. |

**Rationale for WARN-then-RISK escalation:** CONTEXT.md mandates the budget is **OFF by default**
(warn=0, cap=0 → unlimited) and that crossing the **warn** threshold is *non-blocking* while the
**hard cap** *pauses the passive scanner* (interactive chat stays usable either way). The warn state
is purely informational → `Level.WARN` (amber), consistent with Phase 13's survived-secret WARN
banner. The hard-cap state reflects an actively-fired guardrail that paused a subsystem → escalate
to `Level.RISK` (red). This matches the existing project semantics where RISK is reserved for a real
enforced/irreversible-ish condition, not mere advice. `Level.INFO` is too quiet for a budget signal
and is **not** used here.

**Destructive color (`Colors.statusError`):** There are **no destructive/irreversible actions** in
Phase 14. The hard-cap pause is reversible (raise the cap, or it resets next Burp run — budget is
per-process per CONTEXT.md). The RISK banner conveys "a guardrail fired", not a destructive action.
No confirmation dialogs are introduced. (`statusError` is reached only indirectly, via
`SubtleNotice`'s RISK palette = `UiTheme.Colors.accentDanger` = `statusCrashed` = `statusError`.)

## Touch Point 1 — Anthropic Backend Card (BackendConfigPanel)

### Placement

A new private `buildAnthropicPanel(): JPanel` in `ui/panels/BackendConfigPanel.kt`, registered into
the existing `CardLayout` alongside the other backend cards:

```kotlin
cards.add(buildAnthropicPanel(), "anthropic")
```

Add it next to the other HTTP backends (after `buildPerplexityPanel(...)` in the `init` block, around
the existing `cards.add(buildPerplexityPanel(), "perplexity")` line). The card is shown when the user
selects "Anthropic" in the backend selector (the selector itself is existing infrastructure — adding
the new backend id to the dropdown is backend/registry wiring, not a UI-contract concern).

The new fields are declared as panel members exactly like the existing ones, and styled in the same
`applyFieldStyle(...)` block in `init`:

```kotlin
private val anthropicModel = JTextField(initialState.anthropicModel)
private val anthropicApiKey = JPasswordField(initialState.anthropicApiKey)
// ... in init, alongside the other applyFieldStyle calls:
applyFieldStyle(anthropicModel)
applyFieldStyle(anthropicApiKey)
```

`BackendConfigState` gains `anthropicModel: String = ""` and `anthropicApiKey: String = ""`, threaded
through `currentBackendSettings()` (read `String(anthropicApiKey.password).trim()`, exactly like
`perplexityApiKey`) and `applyState(...)` (`anthropicApiKey.text = state.anthropicApiKey`), mirroring
the existing Perplexity/NVIDIA wiring in `SettingsPanel` (`backendState` ↔ `AgentSettings`).

### Anatomy — model this on `buildPerplexityPanel()` exactly

```kotlin
private fun buildAnthropicPanel(): JPanel {
    val panel = formGrid()
    panel.border = EmptyBorder(
        DesignTokens.Spacing.sectionPad, DesignTokens.Spacing.sectionPad,
        DesignTokens.Spacing.sectionPad, DesignTokens.Spacing.sectionPad,
    )
    addRowFull(panel, "Model", anthropicModel)
    addRowFull(panel, "API key (Bearer)", anthropicApiKey)
    addRowFull(panel, "", buildButtonRowPanel(buildTestConnectionButton("anthropic")))
    addSpacerRow(panel, DesignTokens.Spacing.sm)
    return panel
}
```

| Row | Label | Field | Notes |
|-----|-------|-------|-------|
| 1 | "Model" | `anthropicModel` (`JTextField`) | Editable; default value `claude-sonnet-4-6` (see Copywriting + Model-default note). Matches the "Model" row in every other HTTP backend card. |
| 2 | "API key (Bearer)" | `anthropicApiKey` (`JPasswordField`) | Masked input. Persisted **encrypted** via Phase 12 `SecretCipher` (`KEY_ANTHROPIC_API_KEY`). Label string matches the existing backends verbatim for consistency. |
| 3 | (empty label) | `buildButtonRowPanel(buildTestConnectionButton("anthropic"))` | Reuses the existing "Test connection" button builder, id `"anthropic"`. Optional but recommended for parity with OpenAI-compat/NVIDIA/Perplexity. |
| — | trailing spacer | `addSpacerRow(panel, Spacing.sm)` | Identical to every other backend card's trailing spacer. |

**Row set rationale (no Base URL / Extra headers / Timeout rows):** SC1–SC3 require only model
selection + encrypted key + the model-rejection error. The Anthropic Messages API endpoint is fixed
(`https://api.anthropic.com/v1/messages` with the `anthropic-version: 2023-06-01` header — both
backend constants, not user fields). Omitting Base URL / Extra headers / Timeout keeps the card to
the minimum the success criteria demand and avoids implying configurability that does not exist. If
the executor finds a timeout field is needed to route through the shared `CircuitBreaker` defaults,
adding a "Timeout (seconds)" row identical to the other cards is an acceptable, contract-compliant
extension (FLAG-14-02).

### Interaction & validation (from CONTEXT.md — locked)

- **Default model:** the **Model** field defaults to `claude-sonnet-4-6`. Verified against current
  Anthropic docs on 2026-06-10 (see Model-default note below) — it is the current public alias for
  Claude Sonnet 4.6, the balanced Sonnet. The field is editable, so a future alias change is a
  one-line default update and low-risk.
- **API key masking + encryption:** `JPasswordField` masks on screen; the value is encrypted at rest
  via the existing `SecretCipher` pattern on save and decrypted on load. The plaintext value is never
  logged (Phase 12 rule).
- **Model-rejection error (SC3):** a 400 response whose body contains "model" surfaces the **exact**
  user-visible string: **"Anthropic rejected the model ID — check Settings > Anthropic > Model"**
  (see Copywriting). Where this error is surfaced (chat error bubble / existing backend error
  channel) follows the existing error-surfacing path used by `OpenAiCompatibleBackend` — this is a
  backend behavior, not a new UI element; no new dialog or banner is introduced for it.
- **Streaming (SC1) + proxy visibility (SC2):** non-UI — tokens stream through the existing chat
  rendering path; traffic appears in Burp Proxy history because `MontoyaHttpTransport` is used.

### Light/dark compliance

- No `Color(0x…)` / `Font(…)` / spacing literals in the card. `applyFieldStyle` resolves colors from
  `DesignTokens.Colors` at call time — theme-correct on open.
- Use `DesignTokens.Spacing.sectionPad` for the border (not a literal `8`), matching the surrounding
  cards.

## Touch Point 2 — Token-Budget Threshold Inputs (SettingsPanel)

### Placement (recommendation — Claude's discretion per CONTEXT.md, resolved here)

Add a new **"Token budget"** `sectionPanel` **inside the Passive AI Scanner settings region**
(`SettingsPanel.passiveAiScannerSection()`), below the existing scanner controls. Rationale: the
budget's enforcement action is *pausing the passive AI scanner* (CONTEXT.md hard-cap behavior), so the
control belongs adjacent to the subsystem it governs. This keeps the cause (budget thresholds) and the
effect (scanner pause) co-located, and reuses the established multi-`sectionPanel`-per-tab convention
seen in the MCP tab (Phase 10) and the settings tabs (Phase 11).

> **Acceptable alternative (FLAG-14-01):** a standalone "Usage" / "Token budget" section near the
> backend settings or in a dedicated area is equally contract-compliant. The contract recommends the
> Passive-Scanner placement because of the cause/effect locality; the executor may relocate it if the
> tab information architecture reads better otherwise, provided it stays a `sectionPanel` built from
> existing builders.

### Anatomy

```kotlin
sectionPanel(
    title = "Token budget",
    subtitle = "Optional per-session limits. 0 means unlimited (off).",
    content = run {
        val grid = formGrid()
        addRowFull(grid, "Warn threshold (tokens)", warnThresholdField)   // JTextField, integer
        addRowFull(grid, "Hard cap (tokens)", hardCapField)               // JTextField, integer
        addRowFull(grid, "", helpLabel(
            "Warn shows a chat banner. The hard cap pauses passive scanning; chat stays usable."))
        grid
    },
)
```

| Element | Builder / token | Notes |
|---------|-----------------|-------|
| Section wrapper | `sectionPanel(title, subtitle, content)` | From `Components.kt` (existing) |
| Warn-threshold row | `addRowFull(grid, "Warn threshold (tokens)", warnThresholdField)` | `warnThresholdField` is a `JTextField` styled by `applyFieldStyle`; integer-valued |
| Hard-cap row | `addRowFull(grid, "Hard cap (tokens)", hardCapField)` | `hardCapField` is a `JTextField` styled by `applyFieldStyle`; integer-valued |
| Help text | `helpLabel(...)` | Explains off-by-default + pause behavior (≤100 chars; see Copywriting) |

### Interaction & validation (from CONTEXT.md — locked)

- **Defaults:** both fields default to `0`. `0` = unlimited / off (warn=0 AND cap=0 → the guardrail is
  entirely inactive). The guardrail activates only once a user enters a non-zero value (a security
  tool must not surprise-block mid-engagement — CONTEXT.md).
- **Input model:** integers only. Empty input is treated as `0` (off). Non-numeric / negative input
  is coerced to `0` on save (or rejected with no banner change) — the executor follows the existing
  numeric-field handling in `SettingsPanel` (e.g. the existing timeout-seconds fields parse via
  `toIntOrNull()` with a fallback). No new validation-feedback widget is introduced; reuse whatever
  coercion the sibling integer settings already use.
- **Persistence:** two new integer fields on `AgentSettings` (e.g. `tokenBudgetWarnThreshold`,
  `tokenBudgetHardCap`), persisted as plain integers (NOT secrets — not `SecretCipher`). Collected /
  applied through the existing `SettingsPanel` save flow (`collect…` / `apply…`), exactly like the
  other scanner settings.
- **Counting basis (non-UI):** the budget compares combined input+output tokens from `TokenTracker`
  (actual-when-available, else estimate) against the thresholds. This is backend logic; the UI fields
  only carry the two threshold values.
- **No live preview / no per-keystroke validation:** the thresholds take effect on Save like the rest
  of Settings. The chat banner (Touch Point 3) is the only runtime feedback surface.

### Light/dark compliance

- `applyFieldStyle` handles field colors; `helpLabel` handles help-text color (`onSurfaceVariant`).
- No literals. `sectionPanel` provides the section background/border from tokens.

## Touch Point 3 — Token-Budget Banner (ChatPanel)

### Placement

`ui/ChatPanel.kt` currently builds `chatContainer` as a `BorderLayout` with `chatCards`
(`BorderLayout.CENTER`) and `inputPanel()` (`BorderLayout.SOUTH`). **There is no banner today.** Add a
single reused `SubtleNotice` at `BorderLayout.NORTH` of `chatContainer` so it sits above the message
cards and is always visible (not scrolled away) when shown:

```kotlin
private val budgetNotice = SubtleNotice()   // starts hidden
// ... in init, after creating chatContainer:
chatContainer.add(budgetNotice, BorderLayout.NORTH)
chatContainer.add(chatCards, BorderLayout.CENTER)
chatContainer.add(inputPanel(), BorderLayout.SOUTH)
```

`budgetNotice` is a **single instance** whose level + message are updated as the budget state changes;
it is not recreated per message.

### Anatomy

Reuse the existing `SubtleNotice` component (do NOT build a new banner — same stance as Phase 13):

```kotlin
when {
    hardCapReached -> budgetNotice.setMessage(SubtleNotice.Level.RISK, hardCapHtml)   // see Copywriting
    warnCrossed    -> budgetNotice.setMessage(SubtleNotice.Level.WARN, warnHtml)      // see Copywriting
    else           -> budgetNotice.hideNotice()
}
```

| Element | Spec |
|---------|------|
| Component | `SubtleNotice` (existing, `ui/components/SubtleNotice.kt`) |
| Level | `Level.WARN` (warn crossed) → `Level.RISK` (hard cap reached / scanner paused). See Color § for rationale. |
| Visibility | Hidden when no threshold set or neither crossed (`hideNotice()`); component starts hidden |
| Trigger source | `TokenTracker` combined input+output vs the two thresholds (non-UI logic); the banner only reflects the resulting state |
| Behaviour | **Non-blocking.** Interactive chat (input box, Send) stays fully usable in BOTH the WARN and RISK states. The banner never disables Send and never adds a confirmation step. The only enforced effect is the passive-scanner pause (backend), not a chat block. |
| Update thread | Updated on the Swing EDT (the banner lives in `ChatPanel`; `SubtleNotice.setMessage` calls `revalidate()`/`repaint()`). Mutate it only from the EDT, consistent with the rest of `ChatPanel`. |

### Message content

A short one-to-two sentence message with the current usage and the threshold. HTML is acceptable
(`SubtleNotice` auto-wraps plain strings in `<html>` and supports `<b>`). Show the token count and the
threshold so the user understands why the banner appeared; in the RISK state, state that passive
scanning is paused and that chat remains available. Do NOT echo any request content — only token
counts. See Copywriting for exact strings.

### Light/dark compliance

- `SubtleNotice` already overrides `updateUI()` and re-applies its palette from `UiTheme.Colors`
  after a Burp theme switch — no extra handling needed.
- Do not set a background/border on the banner directly; `SubtleNotice` manages its own opacity,
  palette, and padding.

## CAP-03 — Listener-Port Filter (No UI)

CAP-03 adds an optional integer `listener_port` parameter to the MCP `proxy_http_history` tool
(`mcp/tools/`). This is a **tool-schema + handler change with no Burp-panel UI** — the parameter is
consumed by an external MCP client (an AI agent), not rendered in any Swing component. It is listed
here only so the checker and planner know it carries **no UI contract obligations**. Empty/unset →
all ports (current behavior); a port with no matches → empty list, not an error (CONTEXT.md).

## Flags (Non-Blocking)

- **FLAG-14-01:** Token-budget section placement. The contract recommends placing the "Token budget"
  `sectionPanel` inside the Passive AI Scanner settings region (cause/effect locality: the hard cap
  pauses that scanner). A standalone "Usage" / "Token budget" section near the backend settings is an
  equally contract-compliant alternative if the tab IA reads better — provided it remains a
  `sectionPanel` built from existing builders. CONTEXT.md explicitly left placement to Claude's
  discretion.

- **FLAG-14-02:** Anthropic card row set. SC1–SC3 require only Model + encrypted API key (+ the
  reused Test-connection button) — the endpoint and `anthropic-version` are backend constants, so no
  Base URL / Extra headers / Timeout rows are specified. If routing through the shared
  `CircuitBreaker`/timeout defaults turns out to need a user-facing "Timeout (seconds)" field, adding
  one row identical to the other backend cards is acceptable and contract-compliant. Do NOT add a
  Base URL field (the Anthropic endpoint is fixed; a Base URL field would imply false configurability
  and widen the SSRF surface unnecessarily).

- **FLAG-14-03:** Chat banner WARN→RISK escalation. The contract maps warn-crossed → `Level.WARN`
  (amber, advisory) and hard-cap-reached → `Level.RISK` (red, a guardrail fired + scanner paused).
  This differs from Phase 13, which used WARN only, because Phase 14's hard cap is an actively-fired
  enforcement (a subsystem is paused), not pure advice. If the maintainer prefers WARN for both
  states (keeping RISK strictly for the Phase 15 tripwire), downgrading the hard-cap banner to WARN
  is acceptable — but RISK better communicates that scanning has been paused. Both are token-compliant
  (`SubtleNotice` owns the palette).

- **FLAG-14-04:** `ChatPanel` is large (2113 lines) and slated for the Phase 19 mega-file split. This
  phase only **adds** a single `SubtleNotice` member + one `chatContainer.add(...)` line + the
  threshold-state update calls. It MUST NOT introduce new color/font/spacing literals and should keep
  the addition minimal to avoid widening the Phase 19 split surface. The existing chat usage-footer
  (session/global token bars) is unchanged by this phase.
