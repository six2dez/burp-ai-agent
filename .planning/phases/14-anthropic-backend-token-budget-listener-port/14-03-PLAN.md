<objective>
Add an optional `listener_port` integer to the MCP `proxy_http_history` tool (CAP-03, SC5, closes #70): when set, return only proxy-history items received on that Burp listener port; unset → all ports (current behavior); a port with no matches → empty list, not an error. The filter uses the first-class `ProxyHttpRequestResponse.listenerPort()` accessor (verified in the montoya-api 2026.2 jar this project compiles against, RESEARCH §9).

The critical correctness requirement is Pitfall 4: `proxy_http_history` is handled in TWO places in `McpTools.kt` — the `mcpPaginatedTool<GetProxyHttpHistory>` registration (~L649, with `GetProxyHttpHistory` as the lambda receiver) and a manual `decode<GetProxyHttpHistory>` dispatch (~L1860). The filter MUST be added to BOTH `seq` pipelines; adding it to only one leaves the other entry point unfiltered, so the SC5 test must exercise both.

Schema nuance verified during planning (beyond the PATTERNS map): the manual schema map (~L2320) branches on `allowUnpreprocessed` between `GetProxyHttpHistory::class.asInputSchema()` and `GetProxyHttpHistoryRestricted::class.asInputSchema()`. To expose `listener_port` in the tool schema under BOTH settings, the field is added to `GetProxyHttpHistory` (L2719) AND `GetProxyHttpHistoryRestricted` (L2726). The manual decode at L1860 uses `decode<GetProxyHttpHistory>` regardless, so the field is in scope for filtering in both paths.

This plan is code-disjoint from 14-02 (it touches only `McpTools.kt`). It depends on 14-01 only so the full tree compiles cleanly after the AgentSettings additions land (build coherence); it shares no source files with 14-01.

Purpose: Let an external AI agent scope its proxy-history queries to a single Burp listener port (#70).

Output: the `listenerPort` field on the two data classes, the `.filter` in both dispatch paths, and the Wave 0 SC5 test covering both paths + the no-match empty-list + the unset-all-ports cases.
</objective>
