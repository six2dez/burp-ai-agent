<objective>
Ship per-session token-budget guardrails (CAP-04, SC4). The budget decision is a pure AWT-free `BudgetGuard` object (mirrors Phase 13's `SecretShapes`) returning `{OFF, WARN, CAP}` from combined input+output session tokens vs the two thresholds — so SC4 logic is unit-testable without Swing, and `ChatPanel` only renders the enum. The hard cap pauses the passive scanner via a NEW `budgetPaused` gate distinct from `setEnabled` (which clears the knowledge base and flips the user's toggle — wrong for a reversible pause, Pitfall 3). Interactive chat stays usable in every state; the cap pause is per-process (resets on Burp restart).

This plan depends on 14-01 only to read the four `AgentSettings` fields it already declared (`tokenBudgetWarnThreshold`, `tokenBudgetHardCap`, and the Anthropic fields) — 14-01 owns `AgentSettings.kt` to keep the wave-2 plans collision-free. This plan adds the token-budget UI section rendering, declares the two threshold `JTextField`s in `SettingsPanel`, collects them into `AgentSettings`, and threads `passiveAiScanner` into the `ChatPanel` constructor (ChatPanel does not currently hold a scanner reference — PATTERNS Planner Note 6, verified).

Purpose: A cost/DoS guardrail that surfaces consumption and enforces a hard cap without locking the user out mid-engagement — off by default so a security tool never surprise-blocks.

Output: `BudgetGuard.kt` (new), scanner pause gate, chat banner at NORTH, `ChatPanel` constructor threading via `MainTab`, the Settings token-budget section, and Wave 0 tests for SC4a/SC4c (BudgetGuard) + SC4b (scanner pause no-op).
</objective>
