<objective>
Implement PRIV-02 body redaction: reach secrets in request/response bodies (leading form-body field, JSON keys) and apply a user-configurable custom-pattern list — all under the ~1 MB size cap and the `SafeRegex` 50 ms deadline from Plan 01. Persist custom patterns as plaintext config, wire the one-regex-per-line text area + save-time ReDoS validation into the Privacy settings panel, and push compiled patterns into `Redaction` on save.

Purpose: The documented body-redaction gap (the leading `x-www-form-urlencoded` field, missed by the `[?&]`-only `urlTokenParamRegex`) and the lack of any user-extensible redaction are the heart of PRIV-02. AI context payloads are frequently JSON, so JSON-key coverage is included. Custom patterns let users redact org-specific secret shapes without a release.
Output: body/JSON/custom matching in `Redaction.apply`, `Redaction.setCustomPatterns`, `MAX_REDACTION_BODY_CHARS` in `Defaults`, `customRedactionPatterns` persistence in `AgentSettings`, the Privacy panel text-area row + save validation, and the `SettingsPanel` round-trip.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request/response body -> Redaction.apply | Untrusted captured body bytes (form + JSON) cross into the redaction transform; the documented gap is the leading form field |
| User custom regex (Settings) -> SafeRegex -> Redaction | Arbitrary user-supplied regex compiled and run against captured traffic — the ReDoS surface |
| customRedactionPatterns -> Preferences | User config persisted at rest (plaintext — NOT a secret) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-05 | Information Disclosure | Unredacted body secret (leading form field / JSON key) | mitigate | Body-start-anchored `(^|[?&])` form regex + key-scoped JSON regex reach the documented gap; verified by `bodyFormLeadingFieldRedacted` / `bodyJsonSecretKeysRedacted`. ASVS V5. **HIGH — secret leakage.** |
| T-13-06 | Denial of Service | Custom regex catastrophic backtracking during apply OR save | mitigate | Every custom pattern runs via `SafeRegex.replaceAllSafe` (50 ms deadline, fail-open) and is rejected on save via `SafeRegex.isPatternSafe` (compile + ReDoS probe); body stage skipped over the ~1 MB cap. ASVS V5. **HIGH — ReDoS.** |
| T-13-07 | Information Disclosure | Custom patterns persisted / logged | mitigate | Patterns stored PLAINTEXT (config, not secret) — never routed through `SecretCipher`, never added to the SEC-01 migration; redaction never logs matched body content (SafeRegex fail-soft). ASVS V14/V7. |
| T-13-08 | Tampering | Corrupting non-secret body content | mitigate | Key-scoped regexes touch only a value following a sensitive key name; non-sensitive params (`name=alice`, `"name":"alice"`) asserted untouched (Pitfall 3 negative tests). |
| T-13-SC | Tampering | dependency supply chain | accept | Zero new dependencies — all primitives are JDK 21 built-ins (`java.util.regex`, reused `SafeRegex` from Plan 01). RESEARCH Package Legitimacy Audit = not applicable; no slopcheck surface. |
</threat_model>
