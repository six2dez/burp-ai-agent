# Phase 13 — UI Design Contract: Privacy & Redaction Hardening

> Visual and interaction contract for Phase 13 (Privacy & Redaction Hardening).
> Generated by gsd-ui-researcher. Pending verification by gsd-ui-checker.
>
> This contract IS the authoritative design source for the two small, additive UI touch points in
> this phase. All styling MUST come from the Phase 9 design system (`ui/design/DesignTokens.kt` +
> `ui/design/Components.kt`) — no inline `Color()` / `Font()` / spacing literals in panel code.
> The phase is mostly backend work (HKDF, body redaction, custom-pattern engine, `SecretShapes`);
> the UI surface is intentionally minimal and reuses existing components.

---

## Context

**Tech stack:** Kotlin + Swing (JVM 21). No web framework. No shadcn / Tailwind / CSS.
**Platform constraint (ADR-2, locked):** Plain Swing + `javax.swing.UIManager` only. No JavaFX,
no FlatLaf, no third-party L&F. The design system is a Kotlin module, not a CSS file.
**Design system source:** Phase 9 — `DesignTokens.kt` + `Components.kt` in `ui/design/`.
**Light/dark:** Burp switches theme via `UIManager.setLookAndFeel()`. All tokens resolve dynamically
from UIManager keys at call time (computed `get` properties) — never cached.

**Scope of this contract — two touch points only (both additive):**

1. **Custom-regex-pattern input** in `ui/panels/PrivacyConfigPanel.kt` — a one-regex-per-line
   `JTextArea` + an inline validation-feedback label. Validates on Save (regex-compile syntax check
   AND a ~50 ms ReDoS timeout against an adversarial string). Reuses `formGrid` / `addRowFull` /
   `sectionPanel` and the existing `applyAreaStyle` / `helpLabel` builders. (PRIV-02 / PRIV-04)

2. **Survived-secret warning banner** inside the existing redaction preview dialog
   (`ui/components/ContextPreviewDialog.kt`) — a **non-blocking** advisory showing a count and which
   curated secret shape(s) survived post-redaction. Reuses the existing `SubtleNotice` advisory
   component (the same pattern used by `PrivacyConfigPanel`). (PRIV-04)

**Everything else in Phase 13 is non-UI** (HKDF in `Redaction.kt`, body/JSON redaction patterns,
the shared `SecretShapes` curated set, ReDoS timeout primitive, unit tests). The pre-send tripwire
itself (PRIV-03) is **out of scope** — Phase 15.

|------|-------|----------------|

### Validation-feedback color

The inline validation-feedback label below the custom-pattern field is the only Phase-13-specific
color decision. It is reserved for exactly two states and uses semantic tokens:

| Validation state | Token | Copy source |
|------------------|-------|-------------|
| One or more patterns rejected (bad syntax OR ReDoS timeout) | `Colors.statusError` | see Copywriting |
| All patterns valid (shown briefly after a successful Save) | `Colors.statusSuccess` | see Copywriting |
| No patterns entered / not yet validated | hidden (label `isVisible = false`) | — |

This mirrors the existing `saveFeedback` label convention in `PrivacyConfigPanel` (one reusable
label whose text + foreground change with outcome). Do not introduce a new component for it.

### Survived-secret banner color (via `SubtleNotice.Level`)

The banner reuses `SubtleNotice` and picks one of its three existing levels. Phase 13 uses **WARN**:

| Banner state | `SubtleNotice.Level` | Resulting accent (existing tokens) |
|--------------|----------------------|------------------------------------|
| One or more known secret shapes survived redaction | `Level.WARN` | `UiTheme.Colors.subtleWarning` bg + `UiTheme.Colors.accentWarn` accent strip |
| Zero shapes survived | banner hidden (`hideNotice()`) | — |

**Rationale for WARN, not RISK:** CONTEXT.md mandates the indicator is **informational, never a
hard stop** — it reveals what the pipeline missed so the user can decide. RISK (red) is reserved
for the existing privacy-OFF-plus-no-audit advisory and for the Phase 15 tripwire's
warn-with-confirmation. A survived-shape notice is advisory; WARN (amber) is the correct severity.
`Level.INFO` is too quiet for a leaked-secret signal. **Use `Level.WARN`.**

### 60/30/10 Split (unchanged)

- **60% dominant:** `Colors.surface` — both panels' backgrounds.
- **30% secondary:** `Colors.inputBackground` + `Colors.border` — the one new input field.
- **10% accent (`Colors.primary`):** focus ring only in this phase.

**Destructive color (`Colors.statusError`):** reserved for the validation-rejected feedback text
only. There are **no destructive/irreversible actions** in Phase 13 (saving an invalid pattern is
rejected, not destructive; clearing the text area is reversible until Save). No confirmation
dialogs are introduced.

## Touch Point 1 — Custom Redaction Patterns (PrivacyConfigPanel)

### Placement

A new full-width row appended inside the existing `formGrid()` in `PrivacyConfigPanel.build()`,
within the existing `sectionPanel("Privacy & Logging", …)`. Insert it after the existing
"Anonymization" (`rotateSaltBtn`) row and its `addSpacerRow(grid, 4)`, before the "Save feedback"
row — grouping it with the other redaction controls. The field is injected into the panel
constructor (same pattern as `privacyMode`, `rotateSaltBtn`, etc.) and wired from `SettingsPanel`,
matching the existing `privacySection()` construction.

### Anatomy

```
[ JLabel "Custom redaction patterns" ]   [ JTextArea (rows=4, mono, applyAreaStyle) — fills column ]
[ helpLabel("One regex per line. Applied in STRICT and BALANCED. Validated on Save.") ]
[ JLabel validationFeedback — hidden until Save; statusError on reject / statusSuccess on success ]
```

Implement with the existing builders:

| Element | Builder / token | Notes |
|---------|-----------------|-------|
| Row (label + area) | `addRowFull(grid, "Custom redaction patterns", patternsArea, helpText = …)` | `helpText` renders a `helpLabel` on the next grid row automatically |
| Text area styling | `applyAreaStyle(patternsArea)` | Sets `mono` font, `inputBackground`, `inputForeground`, 1 px `border`, `lineWrap`, `wrapStyleWord` |
| Text area rows | `patternsArea.rows = 4` | Matches existing `promptArea` convention |
| Validation feedback | reuse a `JLabel` styled like the existing `saveFeedback` label | `caption` font; foreground swapped per state; `isVisible = false` initially |
| Spacer above/below | `addSpacerRow(grid, DesignTokens.Spacing.xs)` | Matches the 4 px spacers already in this panel |

> If `addRowFull`'s `helpText` parameter is used, the help line is auto-added — do NOT also add a
> separate `helpLabel` row (avoid duplicate help text). The validation-feedback label is a separate
> row appended after the help row.

### Interaction & validation (from CONTEXT.md — locked)

- **Input model:** one regex per line in the `JTextArea`. Blank lines ignored.
- **Validation trigger:** on **Save** (not per-keystroke). Each non-blank line is (a) compiled for
  syntax and (b) run against an adversarial ReDoS test string under a ~50 ms timeout (the same
  timeout primitive the body-redaction path uses — Claude's discretion on mechanism).
- **On any rejection:** the Save does not persist the offending pattern(s); the validation-feedback
  label becomes visible in `statusError` and names how many / which lines failed (see Copywriting).
  Valid lines may still be saved, or the whole save is blocked — executor follows the success
  criterion (SC3: "reject patterns that fail to compile or time out"); rejecting only the offending
  lines while keeping valid ones is acceptable and preferred for usability.
- **On success:** valid patterns persist via the existing config/preference store; the feedback
  label briefly shows the success copy in `statusSuccess` (or stays hidden — both acceptable;
  brief success confirmation preferred for parity with the existing `saveFeedback` behaviour).
- **Replacement token:** matched text is replaced with `[REDACTED]` (consistent with built-ins).
- **Active modes:** STRICT + BALANCED only; inactive in OFF (folded into the `redactTokens` branch).

### Light/dark compliance

- No `Color(0x…)` / `Font(…)` / spacing literals in the panel.
- `applyAreaStyle` resolves colors from `DesignTokens.Colors` at call time — theme-correct on open.
- The validation-feedback label's foreground must be (re)assigned from `Colors.statusError` /
  `Colors.statusSuccess` each time its text is set, so a theme switch between sets is reflected.

## Touch Point 2 — Survived-Secret Banner (ContextPreviewDialog)

### Placement

Inside `ContextPreviewDialog.confirm(...)`, the pre-send preview the user already sees
("Context (as will be sent, after redaction)"). The banner sits **between the header block and the
context body scroll pane** — i.e. added to the existing `header` `BoxLayout(Y_AXIS)` stack after the
"Context (as will be sent, after redaction):" label and before `bodyScroll`, OR as a dedicated strip
just above `BorderLayout.CENTER`. It must be visible without scrolling, since its purpose is to warn
before the user clicks Send.

### Anatomy

Reuse the existing `SubtleNotice` component (do not build a new banner):

```kotlin
val survivedNotice = SubtleNotice()
// ... after computing which curated SecretShapes survived the post-redaction context ...
if (survivedCount > 0) {
    survivedNotice.setMessage(SubtleNotice.Level.WARN, htmlMessage)  // see Copywriting
} else {
    survivedNotice.hideNotice()   // component starts hidden; nothing shown when clean
}
```

| Element | Spec |
|---------|------|
| Component | `SubtleNotice` (existing, `ui/components/`) |
| Level | `Level.WARN` (see Color § for rationale) |
| Visibility | Hidden when zero shapes survive (`hideNotice()`); shown only when `survivedCount > 0` |
| Scan target | **Post-redaction context only** (`contextJson` as it will be sent) — the indicator reveals what survived, not what was caught (CONTEXT.md) |
| Pattern source | The shared `SecretShapes` curated object introduced in this phase (single source of truth, later reused by the Phase 15 tripwire) |
| Behaviour | **Non-blocking.** The dialog keeps its existing `Send` / `Cancel` options unchanged. The banner never disables Send and never adds a confirmation step (that is Phase 15) |

### Message content

A count plus which shape(s) matched, e.g. "2 values matching known secret shapes (OpenAI key,
JWT) survived redaction." HTML is acceptable (`SubtleNotice` wraps plain strings in `<html>`
automatically and bolds via `<b>`). Keep it one to two short sentences. Do not list raw matched
values — name the **shape category** only (the value is already a suspected secret; echoing it in
the banner would re-expose it in the UI). See Copywriting for exact strings.

### Note on the host dialog

`ContextPreviewDialog` is currently **not migrated** to the design system (it uses raw `JLabel` /
`JTextArea`, a hardcoded `BorderLayout(8, 8)`, `Dimension(780, 560)`, and `Box.createVerticalStrut(6)`
literals). Phase 13 only **adds** the `SubtleNotice` banner — a full migration of this dialog to
`DesignTokens` is **out of scope** and explicitly **not required** by this contract. The executor
MUST NOT introduce new literals; the added banner uses `SubtleNotice` (already token-compliant).
Migrating the surrounding dialog is a candidate for a future phase, not Phase 13. (Recorded as
FLAG-13-02.)

### Light/dark compliance

- `SubtleNotice` already overrides `updateUI()` and re-applies its palette from `UiTheme.Colors`
  after a theme switch — no extra handling needed.
- Set `isOpaque` correctly only if wrapping the banner; `SubtleNotice` manages its own opacity.

## States

| State | Where | Presentation |
|-------|-------|--------------|
| Custom patterns — empty / not validated | PrivacyConfigPanel | Text area empty; validation-feedback label hidden |
| Custom patterns — saved valid | PrivacyConfigPanel | Patterns persisted; feedback label shows success copy (`statusSuccess`) briefly, or stays hidden |

There is no "empty data" or "loading" state in this phase — both touch points are inline additions
to panels the user already interacts with.

## Flags (Non-Blocking)

- **FLAG-13-01:** The validation-feedback label reuses the existing `saveFeedback` pattern (one
  `JLabel` whose text + foreground change per outcome). If the executor prefers a dedicated
  `SubtleNotice` for richer multi-line error reporting (e.g. listing each failing line), that is an
  acceptable substitution — both are token-compliant. The contract specifies the simpler label
  because it matches the surrounding panel's established convention and the error copy is short.

- **FLAG-13-02:** `ContextPreviewDialog` is currently un-migrated to `DesignTokens` (raw `JLabel` /
  `JTextArea`, `BorderLayout(8, 8)`, `Dimension(780, 560)`, `Box.createVerticalStrut(6)`). Phase 13
  ADDS a token-compliant `SubtleNotice` banner only; a full dialog migration is **out of scope** and
  not required by any Phase 13 success criterion. Flagged as a candidate for a future UI-debt phase.

- **FLAG-13-03:** `SubtleNotice.Level.WARN` maps (via `UiTheme.Colors.accentWarn`) to `statusTerminal`,
  not to `DesignTokens.Colors.statusWarning`. This is the existing component's palette and is
  intentional — the survived-secret banner inherits whatever `SubtleNotice` WARN renders, keeping it
  visually consistent with the existing privacy/MCP advisories. No new token is introduced.

- **FLAG-13-04:** Whether a rejected Save blocks the entire save or only drops the offending lines is
  left to the executor (SC3 only requires that failing patterns are not accepted). Dropping only the
  invalid lines while persisting the valid ones is preferred for usability; both satisfy the contract.
