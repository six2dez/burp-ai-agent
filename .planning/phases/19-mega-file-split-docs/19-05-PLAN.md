    **Update DECISIONS.md:**

    Append 5 new ADR entries following the exact `## ADR-N: Title` / **Context.** / **Decision.** / **Consequences.** format from PATTERNS.md. Continue the ADR numbering from the last existing entry. Use the exact v0.9.0 decisions from PATTERNS.md Section DECISIONS.md ADR Entries and RESEARCH.md Section 6:

    1. AES-256-GCM secrets at rest (SEC-01):
       - Context: 7+ stored API keys and tokens (backend keys, mcp.token, mcp.tls.keystore.password) were persisted in plaintext in Burp's Preferences. No Bouncy Castle or Tink added (fat-JAR conflict risk).
       - Decision: Encrypt all stored secrets with AES-256-GCM via `javax.crypto`; per-install random 256-bit master key; `ENC1:` prefix for idempotent migration.
       - Consequences: No new runtime dependency; existing secrets migrated on first load; plaintext pass-through preserved for migration compatibility.

    2. Real HKDF host anonymization (PRIV-01):
       - Context: STRICT mode claimed HKDF but used salted SHA-256 (`MessageDigest`). A documented privacy guarantee was not implemented.
       - Decision: Replace `MessageDigest.getInstance("SHA-256")` with `Mac.getInstance("HmacSHA256")` extract/expand (real HKDF); `SecretShapes` becomes the single AWT-free source of truth for privacy-curated patterns.
       - Consequences: Anonymized host values change on upgrade (expected, not a breaking change); existing tests updated to match.

    3. Anthropic backend via MontoyaHttpTransport (CAP-01):
       - Context: Direct Anthropic API access was the top user request. Vendoring the Anthropic SDK would embed its own HTTP client and bypass `MontoyaHttpTransport`, repeating the issue resolved in Phase 7 (#69).
       - Decision: Implement `AnthropicBackend` using the existing `HttpBackendSupport` + `MontoyaHttpTransport` pattern; no vendored SDK; `/v1/messages` API directly over OkHttp routing through Burp.
       - Consequences: Anthropic traffic appears in Burp's Proxy > HTTP history; API key encrypted via SEC-01; native tool-use and prompt-caching deferred to a future phase.

    4. External MCP client untrusted-output trust boundary (CAP-02):
       - Context: External MCP servers may be attacker-controlled or compromised; their tool outputs are untrusted and could contain prompt-injection payloads if fed directly to the AI context.
       - Decision: Wrap all external server tool results in an explicit trust-boundary marker before they enter the AI prompt; kotlin-sdk 0.5.0 (already present) used for SSE/stdio client transport — no SDK bump required (Path A).
       - Consequences: Prompt-injection surface from external servers is bounded; auth tokens stored encrypted (SEC-01); all external tool invocations recorded in the audit log.

    5. Per-session token-budget guardrails (CAP-04):
       - Context: Long passive-scan sessions could exhaust API token budgets silently or unexpectedly.
       - Decision: `BudgetGuard` pure object with reversible CAP/WARN/OFF states; passive scanner pauses when CAP fires; UI warning banner at WARN threshold; 0 = unlimited (off).
       - Consequences: Users can cap spend per session; passive scanner automatically pauses at hard cap; no state leaks between sessions.
