<objective>
Close BUG-69-03: stop the MCP server from leaking Burp HTTP data about hosts the user has not declared in-scope, and stop it from sending traffic to out-of-scope hosts on the AI's request.

Purpose: GitHub issue #69 sub-concern 4 is "MCP returns site_map / proxy_history for EVERY host Burp has seen, including hosts the user explicitly excluded from scope; and `http1_request` will send a request to any URL the AI dreams up". This violates the user's expressed scope boundary. The scanner already has an `activeAiScopeOnly` precedent (passiveAiScopeOnly / activeAiScopeOnly fields) — this plan adds the equivalent for MCP and applies it consistently across every existing scope-aware tool.

Output:
- `AgentSettings.mcpScopeOnly: Boolean = false` + `KEY_MCP_SCOPE_ONLY = "mcp.scope.only"` preference key (no schema bump).
- `McpToolContext.scopeOnly: Boolean = false` so tool handlers see one source of truth.
- `mcp/tools/McpScopeFilter.kt` (new file): two pure helpers, `filterInScope` for read-style tools and `rejectIfOutOfScope` for write-style tools.
- `McpTools.kt`: every tool listed in `must_haves.truths` consults the filter when `ctx.scopeOnly` is true. Tools that ALREADY accept a per-call `scopeOnly` parameter (`proxy_history_annotate` line 689, `response_body_search` line 718) keep that parameter as a per-call override that ORs with the global setting (`effectiveScopeOnly = scopeOnly || ctx.scopeOnly`).
- `McpService.kt`: pass `scopeOnly = settings.mcpScopeOnly` when constructing the per-request McpToolContext.
- `SettingsPanel.kt`: new `Restrict MCP tools to in-scope hosts` checkbox in the MCP section, positioned next to `mcpExternal` / `mcpUnsafe`.
- Three test files: McpScopeFilterTest (unit), McpToolScopeEnforcementTest (per-tool integration), and one new @Test method in AgentSettingsMigrationTest.

OUT OF SCOPE for this plan (deferred per 07-CONTEXT.md):
- Per-tool scope override granularity (today's design is single global toggle — per-tool overrides would be a follow-up if real users ask).
</objective>

            /**
             * Returns items whose URL (as extracted by [urlOf]) is in scope, when ctx.scopeOnly is true.

             * possible, conservative default) and kept when scopeOnly is false. Bytewise no-op when

                urlOf: (T) -> String?,
                ctx: McpToolContext,
            ): Sequence<T>

            /** Convenience overload for List<T> -> Sequence<T>. */
            fun <T> filterInScope(
                items: List<T>,
                urlOf: (T) -> String?,

            /**
             * Returns null when the URL is permitted, or a documented rejection string when scopeOnly is on

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AI agent ↔ MCP server | The AI can call any enabled tool with arbitrary parameters. Today it can read every host Burp has touched and send requests to any URL. The new scope filter restricts both. |
| MCP tool handler ↔ Burp Proxy/SiteMap/Repeater APIs | Tool handlers read from `api.proxy()/api.siteMap()` (privacy boundary: leaks raw URLs/bodies) and write through `api.http()/api.repeater()/api.intruder()` (active-traffic boundary). |
| User-defined scope ↔ MCP enforcement | `api.scope().isInScope(url)` is the canonical source of truth — we never re-implement scope, only consult it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-08 | Information Disclosure | MCP read tools (site_map, proxy_http_history, etc.) | mitigate | Filter results through `api.scope().isInScope` when `ctx.scopeOnly` is true; null URLs are conservatively dropped. Backwards-compatible (default off). |
| T-07-09 | Tampering | MCP write tools (http1_request, repeater_tab, intruder) | mitigate | Short-circuit BEFORE calling `api.http()/api.repeater()/api.intruder()` so out-of-scope URLs never produce outbound traffic, even at the Burp boundary. Test: `verify(api.http(), never()).sendRequest(...)`. |
| T-07-10 | Spoofing | AI sends repeater_tab for an attacker-controlled URL | mitigate | The repeater_tab tool now rejects out-of-scope URLs symmetrically with http1_request, removing a UI-via-AI bypass. |
| T-07-11 | Repudiation | mcpScopeOnly state silently changes | accept | Stored in Preferences like every other AgentSettings field; user can audit via export. The setting itself is boolean — no log entry needed beyond what AgentSettingsRepository already does. |
| T-07-12 | Information Disclosure | WebSocket frame scope check uses upgradeRequest().url() | accept | Best-effort: if the Montoya SDK does not surface the upgrade request from a `WebSocketMessage`, we leave WebSocket tools un-filtered and document the gap in 07-03-SUMMARY for a follow-up. Risk is bounded — WebSocket tools return PAYLOADS, not URLs, so scope leakage is limited to "the AI knows there is a WS connection". |
