<objective>
Close BUG-69-03: stop the MCP server from leaking Burp HTTP data about hosts the user has not declared in-scope, and stop it from sending traffic to out-of-scope hosts on the AI's request.

Purpose: GitHub issue #69 sub-concern 4 is "MCP returns site_map / proxy_history for EVERY host Burp has seen, including hosts the user explicitly excluded from scope; and `http1_request` will send a request to any URL the AI dreams up". This violates the user's expressed scope boundary. The scanner already has an `activeAiScopeOnly` precedent (passiveAiScopeOnly / activeAiScopeOnly fields) — this plan adds the equivalent for MCP and applies it consistently across every existing scope-aware tool.

Output:
- `McpSettings.scopeOnly: Boolean = false` (co-located with `maxBodyBytes`) + `KEY_MCP_SCOPE_ONLY = "mcp.scope.only"` preference key in `AgentSettings.kt`'s companion KEY block. The `mcp.scope.only` preference is read/written by `loadMcpSettings` / `saveMcpSettings` in `AgentSettings.kt`. NO field is added to `AgentSettings` directly — the existing `mcpSettings: McpSettings` sub-object is the single source of truth for MCP-specific knobs (consistent with how `maxBodyBytes`, `unsafeEnabled`, etc. are stored).
- `McpToolContext.scopeOnly: Boolean = false` so tool handlers see one source of truth.
- `mcp/tools/McpScopeFilter.kt` (new file): two pure helpers, `filterInScope` for read-style tools and `rejectIfOutOfScope` for write-style tools.
- `McpTools.kt`: every tool listed in `must_haves.truths` consults the filter when `ctx.scopeOnly` is true. Tools that ALREADY accept a per-call `scopeOnly` parameter (`proxy_history_annotate` line 689, `response_body_search` line 718) keep that parameter as a per-call override that ORs with the global setting (`effectiveScopeOnly = scopeOnly || ctx.scopeOnly`).
- `McpRuntimeContextFactory.kt`: pass `scopeOnly = settings.scopeOnly` when constructing the per-request McpToolContext (`settings` is the existing `McpSettings` parameter to `create(...)`).
- `SettingsPanel.kt`: new `Restrict MCP tools to in-scope hosts` JCheckBox in the MCP section, positioned next to `mcpExternal` / `mcpUnsafe` (mirroring the JCheckBox pattern used by `mcpExternal`/`mcpUnsafe`/`passiveAiScopeOnly`).
- `McpConfigPanel.kt`: new constructor parameter wired into the build() layout (label change for MB → KB lives in plan 07-02; this plan only adds the scope-only row).
- Three test files: McpScopeFilterTest (unit), McpToolScopeEnforcementTest (per-tool integration), and one new @Test method in AgentSettingsMigrationTest.

OUT OF SCOPE for this plan (deferred per 07-CONTEXT.md):
- Per-tool scope override granularity (today's design is single global toggle — per-tool overrides would be a follow-up if real users ask).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AI agent ↔ MCP server | The AI can call any enabled tool with arbitrary parameters. Today it can read every host Burp has touched and send requests to any URL. The new scope filter restricts both. |
| MCP tool handler ↔ Burp Proxy/SiteMap/Repeater APIs | Tool handlers read from `api.proxy()/api.siteMap()` (privacy boundary: leaks raw URLs/bodies) and write through `api.http()/api.repeater()/api.intruder()` (active-traffic boundary). |
| User-defined scope ↔ MCP enforcement | `api.scope().isInScope(url)` is the canonical source of truth — we never re-implement scope, only consult it. |
