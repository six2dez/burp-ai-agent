<objective>
Close BUG-69-01: route every HTTP-based AI backend's healthCheck() through `MontoyaHttpTransport` so Burp's upstream proxy / SOCKS / cert store are honored, and remove the OkHttp fallback from production send() paths so a missing transport fails fast instead of silently bypassing Burp.

Purpose: GitHub issue #69's primary complaint is that AI requests skip Burp's upstream proxy/SOCKS, which breaks the user's mitmproxy/Burp Collaborator setup and contradicts the project's privacy guarantee. The fallback was added defensively but is unreachable from production (AgentSupervisor.kt:70 always constructs a MontoyaHttpTransport and passes it via BackendLaunchConfig); however it's still WIRED, which means a wiring bug would silently bypass Burp. Fail-fast > silent bypass.

Output:
- `MontoyaHttpTransport` already exposes `get()`; no API addition.
- HealthCheck for OpenAi-compatible (covers Perplexity), LM Studio, Ollama, and NVIDIA NIM is routed through `MontoyaHttpTransport`.
- `OpenAiCompatibleBackend.send()` and `LmStudioBackend.send()` lose their `transport == null` OkHttp fallback; production calls always go via Montoya.
- `HttpBackendSupport.buildClient` retains OkHttp for unit tests but its KDoc tells the truth: "OkHttp client for unit tests only; does NOT honor Burp's upstream proxy config".
- `AgentSupervisor.init` enumerates every backend (via the existing `BackendRegistry` API `listAllBackendIds()` + `get(id)` — Option A from the revision notes; no API change needed) and calls `setHealthCheckTransport(httpTransport)` on each HTTP backend instance.
- New `HttpBackendTransportRoutingTest` proves the wiring with mocked `MontoyaApi` / spy `MontoyaHttpTransport`.
</objective>
