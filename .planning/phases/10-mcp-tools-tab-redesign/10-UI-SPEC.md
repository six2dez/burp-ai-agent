# Phase 10 — UI Design Contract: MCP Tools Tab Redesign

> Visual and interaction contract for Phase 10 (MCP Tools Tab Redesign).
> Generated by gsd-ui-researcher. Pending verification by gsd-ui-checker.
>
> This contract IS the authoritative design source for the executor. It supersedes the current
> `buildMcpToolsPanel()` implementation in `SettingsPanel.kt` (line 2166).
> All styling MUST come from the Phase 9 design system — no inline Color/Font/spacing literals.

---

## Context

**Tech stack:** Kotlin + Swing (JVM 21). No web framework. No shadcn / Tailwind / CSS.
**Platform constraint (ADR-2, locked):** Plain Swing + `javax.swing.UIManager` only.
**Design system source:** Phase 9 — `DesignTokens.kt` + `Components.kt` in `ui/design/`.
**Existing component to replace:** `buildMcpToolsPanel()` in `SettingsPanel.kt` (private, ~134 lines).
**Data model (unchanged — UI-07):** `McpToolCatalog.all()` / `available()` / `McpSettings.toolToggles`.
**Build context:** In a store build, `McpToolCatalog.available()` returns only the 8 native tools;
in a full build it returns all 57 tools. The tab handles both cases correctly via the same layout.

Inherited from Phase 9. No new tokens introduced in Phase 10.

## Tab Layout Structure

The redesigned MCP tools tab replaces the return value of `buildMcpToolsPanel()`.
The outer container is a `JScrollPane` from `buildTabPanel(sections)`.

```
buildTabPanel([
  searchBarPanel,           // full-width search field + result count label
  aiToolsSection,           // sectionPanel("AI Tools (extension-native)", subtitle, aiToolsList)
  divider,                  // 1px JSeparator or Spacing.xl rigid area
  montoyaToolsSection,      // sectionPanel("Montoya Tools (generic)", subtitle, montoyaToolsList)
])
```

### Top-level: Search Bar Panel

A `JPanel` with `BoxLayout(X_AXIS)` containing:
- `JTextField` (search input) styled with `applyFieldStyle()` — `Spacing.lg` minimum width, expands horizontally
- `Spacing.sm` horizontal rigid area
- `JLabel` (result count) styled with `helpLabel()` — e.g. "49 tools" / "3 matches"

The search bar panel sits above both sections. It is NOT inside either `sectionPanel`. It has
`EmptyBorder(0, 0, Spacing.lg, 0)` bottom margin before the first section.

### Section 1: AI Tools (extension-native)

```kotlin
sectionPanel(
    title = "AI Tools (extension-native)",
    subtitle = "Extension-native tools — available in both the BApp Store and the full build.",
    content = aiToolsListPanel
)
```

`aiToolsListPanel` is a `JPanel(BoxLayout Y_AXIS)` containing:
1. Bulk-toggle bar (see below)
2. One tool row per tool (sorted alphabetically by `title`)
3. No sub-category grouping — the section is small (8 tools max) and flat layout is clearest

**Tool count in store build:** up to 8 (all native tools are always visible here).
**Tool count in full build:** same 8 (native tools never appear in the Montoya section).

### Section 2: Montoya Tools (generic)

```kotlin
sectionPanel(
    title = "Montoya Tools (generic)",
    subtitle = "Generic Montoya API wrappers — available in the full build only. Not in the BApp Store build.",
    content = montoyaToolsListPanel
)
```

`montoyaToolsListPanel` is a `JPanel(BoxLayout Y_AXIS)` containing:
1. Bulk-toggle bar (see below)
2. Sub-group headers + tool rows, grouped by `McpToolDescriptor.category` (alphabetical category order)

**Sub-grouping within Montoya section:**
Render a category sub-header label before each category group. Sub-header: `JLabel(category)` with
`Typography.label` font + `Colors.onSurfaceVariant` foreground + `EmptyBorder(Spacing.sm, 0, Spacing.xs, 0)`.
Within each category, tools are sorted alphabetically by `title`.

**Montoya categories (current catalog, full build):**
`Burp Control`, `Collaborator`, `Config`, `Editor`, `History`, `Requests`, `Scanner`, `Scope`, `Site Map`, `Utilities`
(10 categories, alphabetical order).

**Store build behavior:** In a store build `McpToolCatalog.available()` returns only native tools,
so `montoyaToolsListPanel` contains zero tools. Render the section with the empty-state label
(see Copywriting) instead of the tool list. The bulk-toggle bar is hidden when the list is empty.

## Tool Row Anatomy

Each tool row is a `JPanel` with `BoxLayout(X_AXIS)`.
Border: `EmptyBorder(Spacing.xs, Spacing.md, Spacing.xs, Spacing.md)`.

Row layout (left to right):
```
[ JCheckBox (title only, no text wrap) ]  [ Spacing.sm gap ]  [ toolBadge ]  [ glue ]  [ (unsafe/pro indicator, if any) ]
[ helpLabel(description) — second sub-row ]
```

Implement as a `JPanel(BorderLayout)` with:
- `BorderLayout.WEST`: `JPanel(BoxLayout X_AXIS)` containing checkbox + gap + badge + glue + optional indicator
- `BorderLayout.SOUTH`: `helpLabel(tool.description)` with `EmptyBorder(0, Spacing.md, 0, 0)` left inset to align under the checkbox

The checkbox text is `tool.title` only — no "(unsafe)" suffix appended to title. The
`unsafeOnly` / `proOnly` state is shown via an indicator label (see below), not by mutating the title.

### Badge Assignment (UI-04)

| `McpToolDescriptor.nativeTool` | Badge label | `BadgeStyle` |
|--------------------------------|-------------|--------------|
| `true` | "Store + Full" | `BadgeStyle.NATIVE` |
| `false` | "Full only" | `BadgeStyle.FULL` |

Call: `toolBadge("Store + Full", BadgeStyle.NATIVE)` or `toolBadge("Full only", BadgeStyle.FULL)`.

### Unsafe / Pro-Only Indicator

Render a small `JLabel` on the right side of the title row for tools with constraints:

| Condition | Indicator text | Font | Foreground |
|-----------|----------------|------|------------|
| `unsafeOnly = true` AND unsafe mode ON | (no indicator — tool enabled normally) | — | — |
| `unsafeOnly = true` AND unsafe mode OFF AND tool allowlisted | "allowlisted" | `Typography.caption` | `Colors.statusWarning` |
| `unsafeOnly = true` AND unsafe mode OFF AND NOT allowlisted | "unsafe" | `Typography.caption` | `Colors.statusError` |
| `proOnly = true` AND edition != Professional | "Pro only" | `Typography.caption` | `Colors.onSurfaceVariant` |
| `proOnly = true` AND edition == Professional | (no indicator) | — | — |

Both `unsafeOnly` and `proOnly` can be true simultaneously — show "Pro only" indicator and
disable the checkbox; the "unsafe" label is not needed because the tool is already disabled.

### Checkbox Enable/Disable Rules (UI-07 — unchanged semantics)

The existing enable/disable logic in `updateUnsafeToolStates()` is preserved exactly:
- `proOnly = true` AND edition != Professional → `isEnabled = false`
- `unsafeOnly = true` AND unsafe mode OFF AND NOT allowlisted → `isEnabled = false`
- `unsafeOnly = true` AND (unsafe mode ON OR allowlisted) → `isEnabled = true`
- All other tools → `isEnabled = true`

Tooltip text: preserve existing `toolTipText` string logic from `buildMcpToolsPanel()` verbatim.

### Checkbox Persistence (UI-07 — unchanged)

Each checkbox is registered in `mcpToolCheckboxes[tool.id] = checkbox` (existing map in `SettingsPanel`).
`collectMcpToolToggles()` and `applyMcpToolToggles()` are NOT changed.
The `McpSettings.toolToggles` map structure is NOT changed.

## Search / Filter (UI-05)

### Behavior

- Filter applies to both sections simultaneously.
- Match predicate: `tool.title.contains(query, ignoreCase = true) || tool.description.contains(query, ignoreCase = true)`
- Filtering is live — a `DocumentListener` on the `JTextField` triggers rebuild or show/hide on each keystroke.
- Clearing the field (empty string) restores all tools in both sections.
- Category sub-headers in the Montoya section are hidden when all tools in that category are filtered out; visible when at least one tool in the category matches.
- The bulk-toggle bar in each section remains visible during filtering if the section has at least one visible tool; hidden if the section has zero visible tools.
- The section itself (`sectionPanel` wrapper) remains visible even with zero results — the empty-state label replaces the tool list content.
- Result count label updates on every filter change: "N tools" (unfiltered) → "N of M tools" (filtered).

### Implementation Approach

Two acceptable implementation approaches — executor chooses:

**Option A (rebuild on filter):** `DocumentListener` calls a `rebuildToolLists(query)` function
that recreates `aiToolsListPanel` and `montoyaToolsListPanel` content, then calls
`revalidate()` + `repaint()` on the parent. Simple but causes brief flicker on fast typing.

**Option B (show/hide rows):** Each tool row and category sub-header is stored in a list.
`DocumentListener` calls `applyFilter(query)` which sets `isVisible` on each row/sub-header.
Calls `revalidate()` + `repaint()` on the list panel. No flicker; slightly more state to maintain.

Both options are compliant with the contract. Option B is preferred for responsiveness.

## Bulk Toggle (UI-05)

### Per-Group Bulk Toggle Bar

Each section (`AI Tools` and `Montoya Tools`) has its own bulk-toggle bar rendered above its tool list.
The bar is a `JPanel(BoxLayout X_AXIS)` with `EmptyBorder(0, 0, Spacing.sm, 0)` bottom margin.

Contents:
```
[ secondaryButton("Enable all") ]  [ Spacing.sm horizontal gap ]  [ secondaryButton("Disable all") ]  [ glue ]
```

### Bulk Toggle Semantics

- "Enable all": sets `isSelected = true` on all ENABLED (not disabled) checkboxes currently VISIBLE
  in that group (respects active search filter). Disabled checkboxes (pro-only, unsafe-locked) are skipped.
- "Disable all": sets `isSelected = false` on all ENABLED checkboxes currently VISIBLE in that group.
- Bulk toggle does NOT call save — it mutates checkbox states in memory. The user saves via the
  existing "Save Settings" flow (UI-07 — no persistence change).
- The existing global `selectAll` / `deselectAll` buttons (lines 2172–2186 in `SettingsPanel.kt`)
  are REMOVED and replaced by the per-group bulk toggles.

### Unsafe Allowlist Section

The existing "Unsafe Allowlist" sub-panel (lines 2233–2271) is preserved as a collapsible
`AccordionPanel` within the Montoya section, below all tool rows.

```kotlin
AccordionPanel(
    title = "Unsafe tool allowlist",
    subtitle = "Approve individual unsafe tools without enabling global unsafe mode.",
    content = allowlistGrid,
    initiallyExpanded = false,
)
```

The allowlist grid layout changes: from `GridLayout(0, 3)` to `BoxLayout(Y_AXIS)` with one
row per tool (same row anatomy as tool rows above, minus the badge). This makes the allowlist
consistent with the redesigned tool rows. The checkbox registration in
`mcpUnsafeApprovalCheckboxes[tool.id]` is unchanged.

## States

### Empty State: Filter Returns No Matches

Shown inside each section's content area when the search filter matches zero tools in that group.

| Element | Spec |

shows the empty state. The bulk-toggle bar is hidden in this state.

When `McpToolCatalog.available()` returns an empty list for the Montoya section in a store build:

| Element | Spec |

## Flags (Non-Blocking)

- **FLAG-10-01:** The `toolBadge` painter uses `bgColor` captured at creation time (not a `get`
  property). If Burp switches themes at runtime, existing badge instances keep the stale color until
  the panel is rebuilt. Mitigation: override `updateUI()` on the badge `JLabel` subclass to
  reapply the `bgColor` / `fgColor` from the current `DesignTokens.Colors` values. The executor
  should implement this if dynamic theme switching is a stated user expectation; otherwise the
  next panel open after a theme switch will have correct colors.

- **FLAG-10-02:** The category sub-header for the Montoya section uses an inline `JLabel` with
  manual token references (`Typography.label`, `Colors.onSurfaceVariant`) rather than a dedicated
  builder. This is intentional — adding a `categoryHeader()` builder to `Components.kt` would
  be premature generalization for a single usage point. Phase 11 may extract it if needed.

- **FLAG-10-03:** The `ScoreOrder` for Montoya categories is alphabetical by category name. The
  current catalog has `Requests` (13 tools) as the largest category. If the user finds alphabetical
  ordering unhelpful, a custom sort order can be introduced in Phase 10 plans. The contract leaves
  ordering to the executor since no user preference was expressed.

- **FLAG-10-04:** The "Reset to defaults" button (Phase 9 Copywriting) was defined for Phase 11's
  settings rollout. It is out of scope for Phase 10's MCP tools tab. The bulk-toggle approach
  ("Enable all" / "Disable all") covers the reset-to-defaults use case for tool toggles.
